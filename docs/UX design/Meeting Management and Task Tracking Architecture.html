1. Product Summary & Goals

Product Name: MeetFlow

Purpose: A web and mobile-responsive application designed to streamline meeting management, agenda tracking, task assignment, and automated follow-ups. MeetFlow helps teams ensure accountability by tracking open tasks across recurring meetings and sending automated WhatsApp reminders to responsible individuals.

Target Users:

  • Team leaders and managers
  • Project coordinators
  • Small to medium-sized teams (5-50 members)
  • Organizations requiring structured meeting follow-through

Key Value Propositions:

  • Never lose track of meeting action items with automatic task carry-forward
  • Clear accountability through responsible person assignment
  • Automated WhatsApp reminders ensure tasks are completed
  • Hierarchical recurring meetings with visual progress tracking
  • Simple, intuitive interface focused on productivity

5 Core Features:

  1. User Access & Team Management - Secure authentication and centralized team contact repository
  2. Meeting Scheduling & Dashboard - Create meetings and monitor progress at a glance
  3. Agenda & Task Management - Structured agenda points with status tracking and responsibility assignment
  4. Task Continuity & Reminder System - Automatic carry-forward of open tasks and WhatsApp-based reminders
  5. Recurring & Follow-up Meetings - Parent meetings with hierarchical follow-up sessions and collapsed UI for clarity

2. System Architecture

2.1 Recommended Tech Stack

Layer Technology Rationale
Frontend Next.js 14 (App Router) Modern React framework with SSR, mobile-responsive, excellent DX
Backend Node.js + Express Lightweight, fast, excellent ecosystem for REST APIs
Database MongoDB + Mongoose Flexible document model perfect for hierarchical meetings structure
Authentication JWT (Access + Refresh tokens) Industry standard, stateless, secure with httpOnly cookies
Messaging Meta WhatsApp Business Cloud API Free tier available, reliable, widely used platform
Background Jobs node-cron Simple scheduler for daily reminder jobs, no Redis dependency
Hosting Railway.app or Render.com Cost-effective, easy deployment, built-in MongoDB hosting

2.2 Architecture Overview

High-Level Flow:

┌─────────────┐ │ Client │ (Next.js - Web/Mobile Browser) │ (Browser) │ └──────┬──────┘ │ HTTPS/REST ▼ ┌─────────────────────────────────────┐ │ Express.js REST API │ │ ┌─────────────────────────────┐ │ │ │ Routes → Controllers │ │ │ │ Controllers → Services │ │ │ │ Services → Models │ │ │ └─────────────────────────────┘ │ └──────┬──────────────────┬───────────┘ │ │ ▼ ▼ ┌──────────────┐ ┌─────────────────┐ │ MongoDB │ │ Cron Scheduler │ │ Database │ │ (Daily 8 AM) │ └──────────────┘ └────────┬────────┘ │ ▼ ┌────────────────────┐ │ WhatsApp Cloud API│ │ (Send Reminders) │ └────────────────────┘

Architectural Pattern: Layered Architecture

  1. Routes Layer - HTTP endpoint definitions, request validation
  2. Controllers Layer - Request/response handling, business logic orchestration
  3. Services Layer - Core business logic, data transformations, external API calls
  4. Models Layer - Database schema definitions and data access

3. Database Schema Design

3.1 Collection: users

Purpose: Store user account information for authentication

Field Type Constraints Description
_id ObjectId Primary Key Auto-generated unique identifier
email String Required, Unique, Lowercase, Indexed User's email address for login
passwordHash String Required Bcrypt hashed password (10 rounds)
name String Required User's full name
createdAt Date Auto-generated Account creation timestamp
updatedAt Date Auto-generated Last update timestamp

Indexes: email (unique, ascending)

3.2 Collection: teamMembers

Purpose: Store team contact information for task assignment and WhatsApp notifications

Field Type Constraints Description
_id ObjectId Primary Key Auto-generated unique identifier
userId ObjectId Required, Indexed, Ref: users Owner of this contact
name String Required Team member's full name
phone String Required Phone in E.164 format (e.g., +971501234567)
createdAt Date Auto-generated Record creation timestamp
updatedAt Date Auto-generated Last update timestamp

Indexes: userId (ascending)

3.3 Collection: meetings

Purpose: Store all meetings including standalone, parent (recurring), and follow-up sessions

Field Type Constraints Description
_id ObjectId Primary Key Auto-generated unique identifier
userId ObjectId Required, Indexed, Ref: users Meeting owner
title String Required Meeting title/name
date Date Required, Indexed Scheduled meeting date
type String Required, Enum 'standalone', 'parent', or 'followup'
parentMeetingId ObjectId Optional, Indexed, Ref: meetings Parent meeting reference (only for followup type)
followupSequence Number Optional Sequential number (1,2,3...) for follow-ups
recurrence Object Optional Recurrence settings (only for parent type)
status String Required, Enum, Indexed 'upcoming', 'active', 'completed'

Indexes: userId, parentMeetingId, date, status

3.4 Collection: agendaPoints

Purpose: Store agenda items/tasks for meetings with status tracking and responsibility assignment

Field Type Constraints Description
_id ObjectId Primary Key Auto-generated unique identifier
meetingId ObjectId Required, Indexed, Ref: meetings Associated meeting
userId ObjectId Required, Ref: users Agenda point creator
description String Required Task/agenda item description
responsiblePersonId ObjectId Optional, Indexed, Ref: teamMembers Assigned team member
status String Required, Enum, Default: 'open' 'open' or 'closed'
carriedForwardFromId ObjectId Optional, Ref: agendaPoints Origin task if carried forward

4.3 Meetings Endpoints

Base Path: /api/meetings

GET / - List all meetings with optional filters

POST / - Create new meeting

{
  "title": "Weekly Team Sync",
  "date": "2026-03-10T09:00:00.000Z",
  "type": "parent",
  "recurrence": {
    "enabled": true,
    "frequency": "weekly",
    "dayOfWeek": 1
  }

GET /:id - Get meeting details with populated agenda points

PUT /:id - Update meeting details

DELETE /:id - Delete meeting (cascades to delete agenda points)

GET /:parentId/followups - List all follow-up meetings under a parent

POST /:parentId/followups - Create next follow-up meeting (auto-carries forward open tasks)

GET /dashboard - Get dashboard view with aggregated meeting statistics

4.4 Agenda Points Endpoints

Base Path: /api/meetings/:meetingId/agenda

GET / - List all agenda points for a meeting

POST / - Create new agenda point

{

DELETE /:agendaId - Delete agenda point

Base Path: /api/reminders

GET /preview/:meetingId - Preview what reminders would be sent

5. Key Business Logic

5.1 Carry Forward Logic

Trigger: When creating a new follow-up meeting via POST /api/meetings/:parentId/followups 

FUNCTION createFollowUpWithCarryForward(parentId, optionalDate):
  
  1. Validate parent meeting exists and type = 'parent'
  
  2. Find last follow-up:
     lastFollowup = Meeting.findOne({
       parentMeetingId: parentId,
       type: 'followup'
     }).sort({ followupSequence: -1 })
  
  3. Calculate new sequence:
     newSequence = lastFollowup ? lastFollowup.followupSequence + 1 : 1
  
  4. Create new meeting document
  
  5. Find open tasks from last follow-up:
     IF lastFollowup exists:
       openTasks = AgendaPoint.find({
         meetingId: lastFollowup._id,
         status: 'open'
       })
  
  6. Clone open tasks into new meeting with carriedForwardFromId reference

5.2 WhatsApp Reminder Flow

Trigger: Cron job runs daily at 8:00 AM

FUNCTION sendDailyReminders():
  
  1. Find upcoming meetings for tomorrow
  
  2. For each meeting, find open agenda points with responsible persons
  
  3. Group tasks by responsible person
  
  4. For each person:
     a. Build message string with list of tasks
     b. Send via Meta WhatsApp Cloud API
     c. Log result

6. Frontend Screen Map

Screen 1: Login / Register

User authentication screens. On login, store JWT in memory and refresh token in httpOnly cookie.

Screen 2: Dashboard

Overview of all meetings. Cards display title, date, and status. Visual indicators: Green badge for recurring, checkmark for closed tasks, warning icon for open tasks.

Screen 3: Meeting Detail Page

Manage single meeting agenda. Inline form to add points. Drag-and-drop to reorder. Toggle status (open/closed). "Send Reminders" button at top.

Screen 4: Parent Meeting & Follow-ups

Hierarchical view. Accordion layout for follow-up sessions. Latest session expanded by default. "Schedule Next Follow-up" button triggers carry-forward logic.

Screen 5: Team Members Page

Directory of contacts. Simple table to add, edit, or delete team members.

8. Security Considerations

  • Password Security: Bcrypt with 10 salt rounds.
  • JWT Strategy: Access Token (15m lifetime) in memory, Refresh Token (7d lifetime) in httpOnly cookie.
  • Data Isolation: All database queries must include userId to prevent data leakage between tenants.
  • Input Validation: Joi or express-validator for all incoming request bodies.
  • Rate Limiting: Implement on auth routes to prevent brute force.
  • CORS: Restrict to frontend domain only.
